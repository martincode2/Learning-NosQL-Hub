Introduction to NoSQL Databases

What is NoSQL?

NoSQL (Not Only SQL) refers to a class of databases that go beyond the traditional relational database model. They are designed to handle large volumes of unstructured data and are optimized for distributed systems.

NoSQL databases emerged in the late 2000s as web companies faced challenges in handling the massive amounts of data generated by their users. Unlike relational databases, NoSQL databases don't require a fixed schema, and they can scale horizontally across many servers.

Why NoSQL?

NoSQL databases offer several advantages over traditional relational databases:

  • Scalability: NoSQL databases are designed to scale out by distributing data across many servers
  • Flexibility: NoSQL databases typically have flexible schemas that can evolve as needs change
  • Performance: NoSQL databases are optimized for specific data models and access patterns
  • High Availability: Many NoSQL databases are designed for high availability with no single point of failure

Key Terms and Definitions

NoSQL (Not Only SQL)

A class of databases that go beyond the relational database model. Designed to handle large amounts of unstructured data and distributed systems. Uses non-relational data structures: documents, key-value stores, graph databases, wide-column stores.

MongoDB

Popular NoSQL database using document-oriented model. Stores data in flexible JSON-like documents. Features: automatic sharding, replication, and indexing.

Types of NoSQL Databases

Document Databases

Store data as documents (JSON/BSON format) in collections. Ideal for content management, user profiles, and catalogs.

MongoDB

Popular document database used by eBay, Forbes, and Adobe

Key-Value Stores

Simple key-value pairs. Extremely fast, perfect for caching, session storage, and leaderboards.

Redis

In-memory key-value store used by Twitter, GitHub, and Snapchat

Column-Family Stores

Store data in columns rather than rows. Optimized for queries over large datasets and write-heavy workloads.

Cassandra

Distributed column-family database used by Netflix, Instagram, and Spotify

Graph Databases

Store data as nodes and relationships. Perfect for social networks, recommendation engines, and fraud detection.

Neo4j

Graph database used by NASA, Walmart, and Cisco

Data Formats: JSON, BSON, and XML

JSON (JavaScript Object Notation)

Human-readable, text-based, lightweight. Commonly used in APIs and web applications.

{
  "name": "Alice",
  "age": 25,
  "skills": ["Python", "MongoDB"]
}

BSON (Binary JSON)

Binary-encoded form of JSON, used in MongoDB. Supports extra data types (e.g., Date, ObjectId). Faster and more efficient for storage/retrieval, but not human-readable.

XML (eXtensible Markup Language)

Text-based, uses tags to describe data. More verbose, widely used in legacy systems and configurations.

<person>
  <name>Alice</name>
  <age>25</age>
  <skills>
    <skill>Python</skill>
    <skill>MongoDB</skill>
  </skills>
</person>

Database Requirements and Analysis

Identifying Database Requirements

Database requirements are derived from user requirements and form the foundation for database design and implementation.

Key Methods for Requirement Identification:

  • User Research: Interviews, surveys, observations, focus groups
  • Personas: Detailed fictional characters representing target users
  • User Stories: Requirements in narrative format (INVEST criteria)
  • Requirements Gathering: Brainstorming, prototyping, wireframing, mockups
  • User Testing: Usability testing, feedback gathering, iteration

Use Cases Definition

Key Components of Use Cases:

  • Actor: Person or entity interacting with system
  • Goal: Objective the actor wants to achieve
  • Precondition: Required conditions before use case begins
  • Post condition: Expected conditions after completion
  • Flow of events: Step-by-step interaction description

Steps for Defining Use Cases:

  1. Identify actors
  2. Identify goals
  3. Create use cases
  4. Write detailed descriptions
  5. Refine and prioritize

Requirements Analysis Process

Definition: Critical phase in software development involving identifying, documenting, and validating specific system needs.

Key Steps:

  1. Identify Key Stakeholders and End-Users: Determine all involved parties, consider different perspectives, understand needs, goals, expectations, constraints
  2. Capture Requirements: Use various techniques: interviews, surveys, workshops. Employ tools: mind maps, user stories, use cases. Utilize visual representation methods.
  3. Categorize Requirements: Group similar requirements, prioritize based on importance and impact, consider constraints (budget, timeline, technical limitations)
  4. Interpret and Record Requirements: Document requirements clearly and concisely, use consistent format and standardized templates, define all terms and acronyms
  5. Validate Requirements: Review with stakeholders, address concerns and incorporate feedback, conduct feasibility studies, use traceability matrix and walkthroughs

Data Analysis Process

Steps for Performing Data Analysis:

  1. Define the Problem and Research Question: Focus on business value-driven analysis, avoid unnecessary data collection
  2. Collecting Data: First-party data (from customers/audiences on owned channels), second-party data (shared between organizations), third-party data (purchased from external sources)
  3. Preparing Customer Data: Remove errors and duplicates, fill missing data, structure for easier manipulation, remove irrelevant data
  4. Analyzing the Data: Descriptive analysis (past events and patterns), diagnostic analysis (root cause understanding), predictive analysis (future trends forecasting), prescriptive analysis (recommendations for actions)
  5. Interpret the Results: Clarify research questions, draw meaningful conclusions

Data Validation Implementation

Validation Steps:

  • Range Checking: Verify values within predefined limits
  • Format Checking: Validate specific formats (email, dates, phone numbers)
  • Consistency Checking: Ensure related data values are consistent
  • Completeness Checking: Verify all required fields are filled
  • Uniqueness Checking: Ensure certain fields have unique values
  • Cross-Validation: Compare data from multiple sources
  • Business Rule Validation: Check adherence to predefined rules
  • Regular Expression Validation: Use patterns for specific validations
  • Data Type Validation: Verify correct data types
  • Data Quality Checks: Evaluate accuracy, completeness, consistency, timeliness

Preparing Database Environment

MongoDB Scalability Factors

Horizontal Scalability:

  • Sharding: Distribute data across multiple servers
  • Automatic Sharding: Based on predefined sharding keys
  • Balanced Sharding: Even data distribution across shards
  • Indexing: Speed up query execution

Vertical Scalability:

  • Memory Optimization: Efficient memory usage
  • Query Optimization: Choose efficient execution plans

Additional Factors:

  • Replication: Data redundancy and fault tolerance
  • High Availability: Replica sets and sharded clusters
  • Cloud Integration: Available on various cloud platforms

Setting Up MongoDB Environment

Shell Environment Setup - Installation Steps:

  1. Visit official MongoDB website (https://www.mongodb.com/)
  2. Download appropriate version for your operating system
  3. Follow installation instructions
  4. Set environment variables (add MongoDB bin to PATH)
  5. Start MongoDB server: mongod
  6. Connect to MongoDB shell: mongo

Windows Installation Process:

  1. Download MongoDB installer
  2. Run setup file and click Next
  3. Choose Complete installation
  4. Select "Run services as Network service user"
  5. Optionally install MongoDB Compass (GUI)
  6. Complete installation
  7. Open Command Prompt
  8. Navigate to: C:\Program Files\MongoDB\Server\4.0\bin
  9. Run: mongod to start server
  10. Run: mongo.exe to start client

MongoDB Environments

Shell Environment (Mongosh)

Command-line interface for MongoDB. Direct database interaction. Suitable for developers and administrators.

Compass Environment

Official MongoDB GUI. Visual database management. User-friendly interface for non-technical users. Data visualization and query building tools.

Atlas Environment

Cloud-based MongoDB service. Automated deployment and scaling. Managed database solution. Global reach and high availability.

Database Environment Setup Steps

General Process:

  1. Choose a NoSQL Database: Evaluate needs (scalability, performance, consistency), research options (MongoDB, Cassandra, Redis)
  2. Install and Configure the Database: Download and install following vendor instructions, configure settings (storage, memory, network)
  3. Create Database and Collections: Database: Logical container for data, Collections: Groups of related documents
  4. Connect to the Database: Use appropriate driver for programming language, establish connection with connection string

Real-World Examples and Use Cases

MongoDB at eBay

eBay uses MongoDB for numerous projects including search suggestions, metadata storage, and cloud management. MongoDB's flexible data model allows eBay to handle diverse data types and evolve their applications quickly.

// Example of product data in eBay's MongoDB
{
  _id: ObjectId("5f9d9b9d9c9b9a9b9c9d9e9f"),
  product_id: "123456789",
  title: "Vintage Coffee Maker",
  price: { value: 49.99, currency: "USD" },
  seller: { id: "seller123", rating: 4.8 },
  tags: ["vintage", "kitchen", "coffee"],
  created_at: ISODate("2023-10-27T10:00:00Z")
}

Redis at Twitter

Twitter uses Redis to cache timelines and support real-time delivery of tweets. Redis's in-memory nature provides extremely fast read and write operations, essential for real-time social media feeds.

// Example of timeline data in Twitter's Redis
HSET timeline:user123 tweet:789 "Hello world!"
HSET timeline:user123 tweet:790 "Having a great day!"
HGETALL timeline:user123
> 1) "tweet:789"
> 2) "Hello world!"
> 3) "tweet:790"
> 4) "Having a great day!"

Cassandra at Netflix

Netflix uses Cassandra for its scalability and fault-tolerance, storing user viewing data and recommendations. Cassandra provides high availability with no single point of failure, crucial for Netflix's global streaming service.

-- Example of user data in Netflix's Cassandra
CREATE TABLE user_watch_history (
  user_id UUID,
  movie_id UUID,
  watch_time TIMESTAMP,
  progress INT,
  rating INT,
  PRIMARY KEY (user_id, watch_time)
) WITH CLUSTERING ORDER BY (watch_time DESC);

Neo4j at NASA

NASA uses Neo4j to store and analyze data about the International Space Station. The graph database helps model complex relationships between astronauts, experiments, equipment, and spacewalks.

// Example of data relationships in NASA's Neo4j
MATCH (astronaut:Astronaut)-[:PARTICIPATED_IN]->
(experiment:Experiment)<-[:USED_IN]-(equipment:Equipment)
WHERE experiment.name = "Microgravity Plant Growth"
RETURN astronaut.name, equipment.type;

NoSQL vs SQL Comparison

Feature SQL Databases NoSQL Databases
Data Model Relational (Tables) Non-relational (Documents, Key-Value, etc.)
Schema Fixed, predefined Flexible, dynamic
Scaling Vertical (hardware upgrade) Horizontal (adding more servers)
ACID Compliance Yes Often relaxed (BASE model)
Use Cases Complex queries, transactions Big data, real-time apps
Examples MySQL, PostgreSQL, Oracle MongoDB, Cassandra, Redis

When to Use NoSQL

Consider using a NoSQL database when:

  • You need to scale horizontally
  • Your data schema is fluid or changing
  • You're dealing with large volumes of data
  • Your application requires high availability
  • Your data is unstructured or semi-structured
  • You need high throughput for read and write operations

When to Use SQL

Consider using a SQL database when:

  • You need ACID compliance for transactions
  • Your data structure is stable and well-defined
  • You need complex queries and joins
  • You have relational data with integrity constraints
  • Your team has more experience with relational databases
